# План: автоматический мониторинг Instagram → Google Sheets

## Цель

Посты с исследованиями из отслеживаемых аккаунтов автоматически попадают в таблицу по мере выхода в Instagram. Только экспертный контент; личные, маркетинговые и рекламные посты пропускаются. Поддержка нескольких аккаунтов.

---

## 1. Хранение настроек и состояния

**Файл состояния:** `~/.ig_parser_state.json` (или `data/sync_state.json` в проекте)

```json
{
  "last_run_at": "2026-02-26T12:00:00Z",
  "accounts": ["dangarnernutrition", "account2", "account3"],
  "processed_post_ids": ["post_id_1", "post_id_2"]
}
```

- `last_run_at` — время последней успешной проверки (для `onlyPostsNewerThan`)
- `accounts` — список аккаунтов для мониторинга
- `processed_post_ids` — ID уже обработанных постов (дедупликация)

**Альтернатива:** SQLite-таблица или Redis, если позже нужна более сложная логика.

---

## 2. Фоновый процесс (scheduler)

**Варианты:**

### A. Cron (рекомендуется для VPS)

- Отдельный скрипт `scripts/sync_worker.py` — читает state, вызывает пайплайн, обновляет state.
- Запуск **раз в сутки в 8:00 по Москве** (экономия токенов Apify/OpenAI).

**Crontab (8:00 МСК = 5:00 UTC):**
```bash
0 5 * * * cd /root/IG_parser && /root/.local/bin/poetry run python scripts/sync_worker.py
```

### B. Тестовый режим: загрузка по кнопке

- В Streamlit UI — кнопка «Запустить синхронизацию» для ручного запуска.
- Подключение тестового аккаунта для проверки сбора материала перед включением cron.

Для VPS: вариант A (cron). На этапе отладки — вариант B (кнопка).

---

## 3. Инкрементальная выборка постов

**Apify поддерживает `onlyPostsNewerThan`.** В `apify_service.py`:

- Добавить параметр `only_posts_newer_than: str | None = None` в `fetch_posts()`.
- В `_build_posts_input()` добавить:

  ```python
  if only_posts_newer_than:
      input["onlyPostsNewerThan"] = only_posts_newer_than
  ```

- При каждом запуске передавать, например, `"1 day"` или дату последнего запуска в ISO.

**Дедупликация:** хранить `id` обработанных постов в state и пропускать их при повторной обработке.

---

## 4. Фильтр «только экспертные посты»

### Этап 1: правила (без AI)

Пропускать пост, если:

- нет PMID и нет совпадений с EVIDENCE_TERMS (`pmid`, `pubmed`, `research`, `study`, `meta-analysis`, `position stand`, `ISSN` и т.п.);
- есть явные маркеры рекламы: `#реклама`, `#ad`, `#партнёрство`, `sponsored`, `paid partnership`;
- пост слишком короткий (например, < 50 символов без хэштегов).

Пропуск по правилам делается в пайплайне до тяжёлой обработки (PubMed, картинки).

### Этап 2: AI-классификация (по желанию)

Для «пограничных» постов:

- промпт: «Этот пост про научное исследование / экспертизу или про личный опыт / маркетинг / рекламу?»;
- ответ: `research` / `skip`;
- при `skip` — пост не обрабатывать.

Можно сделать опциональным (env-флаг `USE_AI_POST_FILTER=true`).

## 5. Поддержка нескольких аккаунтов

**Сейчас:** `sources` уже передаётся как список, Apify принимает `username: [list]`.

**Нужно:**

1. **UI (Streamlit):**
   - Поле «Аккаунты для мониторинга» — text area или multiselect.
   - Список сохраняется в state (файл/БД) при каждом запуске воркера или через кнопку «Сохранить настройки».

2. **Воркер:**
   - Читает список аккаунтов из state.
   - Вызывает `fetch_posts(sources=accounts, ...)` с этим списком.

## 7. Порядок реализации

1. **sync_state.py** — чтение/запись state.
2. **apify_service** — поддержка `onlyPostsNewerThan`.
3. **pipeline** — ранний фильтр не-исследовательских постов (правила).
4. **sync_worker.py** — скрипт, вызывающий пайплайн и обновляющий state.
5. **streamlit_app** — UI для списка аккаунтов и сохранения настроек.
6. **pipeline** — опциональный AI-фильтр (если нужен).
7. **deploy** — cron (8:00 МСК) и документация.

---

## 8. Этапы внедрения

1. **Тест по кнопке** — подключить тестовый аккаунт, проверять сбор материала вручную через кнопку «Запустить» в Streamlit.
2. **Автоматизация** — после проверки настроить cron на 8:00 МСК.

---

## 9. Риски и ограничения

- **Apify квоты** — запуск раз в сутки (8:00 МСК) минимизирует расход токенов.
- **PubMed 429** — сохраняются текущие ограничения (задержки, лимиты).
- **Дубликаты** — нужна дедупликация по `post_id` или `post_url`, чтобы один пост не попадал в таблицу дважды.
